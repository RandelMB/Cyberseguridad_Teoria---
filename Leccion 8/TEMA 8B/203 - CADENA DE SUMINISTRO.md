Las vulnerabilidades de la cadena de suministro de software se refieren a los riesgos y debilidades potenciales introducidos en los productos de software durante su ciclo de vida de desarrollo, distribución y mantenimiento. Esta cadena de suministro describe muchas etapas, desde la codificación inicial hasta la implementación por parte del usuario final, e incluye varios proveedores de servicios, de hardware y de software.

Proveedores de servicios
Los proveedores de servicios, como los servicios en la nube o las agencias de desarrollo de terceros, cumplen una función en la cadena de suministro de software al ofrecer plataformas de desarrollo, pruebas e implementación, o para contribuir directamente a la base de código del software. Se pueden introducir vulnerabilidades si estos servicios tienen medidas de seguridad inadecuadas o si la comunicación entre estos servicios y el resto de la cadena de suministro no está asegurada correctamente.

Proveedores de hardware
Los proveedores de hardware desempeñan un papel fundamental en la cadena de suministro de software y pueden ser fuentes potenciales de vulnerabilidades. El hardware en el que se ejecuta o interactúa el software es la base de los recursos tecnológicos. Si esta capa de hardware se ve comprometida, puede provocar graves problemas de seguridad. Esto es sobre todo cierto para el firmware o los controladores de software de bajo nivel que interactúan estrechamente con el hardware. Si un proveedor de hardware no aplica prácticas de seguridad sólidas en sus procesos de diseño y fabricación, puede introducir vulnerabilidades que comprometan todo el sistema.

Por ejemplo, un componente de hardware podría venir con un firmware preinstalado que contuviera una vulnerabilidad conocida, o podría ser susceptible a la manipulación física que condujera a una violación de la seguridad. Del mismo modo, los dispositivos de hardware a menudo requieren controladores específicos para funcionar correctamente. Si estos controladores no se actualizan de manera regular o se obtienen de proveedores poco confiables, pueden introducir vulnerabilidades en el software.

Además, los proveedores de hardware a menudo proporcionan toda el software que se ejecuta en el dispositivo para IoT y sistemas integrados, lo que los convierte en un factor importante para la seguridad general de un sistema. Por lo tanto, es fundamental para la seguridad de la cadena de suministro de software garantizar que los proveedores de hardware cumplan con los estrictos estándares de seguridad, y que los componentes del hardware, así como el software de bajo nivel asociado se actualicen y parcheen regularmente a fin de abordar cualquier vulnerabilidad potencial.

Proveedores de software
Los proveedores de software, incluidos los fabricantes de bibliotecas, entornos de trabajo (frameworks) y otros componentes de terceros utilizados en el software, son una fuente común de vulnerabilidades. Si los componentes de terceros tienen vulnerabilidades o están desactualizados, pueden exponer toda la aplicación a posibles ataques. En todas estas relaciones, la confianza se deposita implícitamente en cada proveedor para mantener una alta seguridad. Si algún eslabón de esta cadena no cumple con estas expectativas, puede provocar vulnerabilidades en la cadena de suministro de software. 

Lista de materiales del software
Una lista de materiales de software (SBOM) es un inventario completo de todos los componentes de un producto de software. Incluye el código de la aplicación principal y todas las dependencias, como las bibliotecas, los entornos de trabajo y otros componentes de terceros. El SBOM incluye detalles como los nombres de componentes, las versiones e información sobre los proveedores.

Un SBOM tiene como objetivo proporcionar transparencia y visibilidad en la cadena de suministro de software, lo que puede ayudar significativamente a mitigar los problemas de la cadena de suministro de software. Al detallar todos los componentes utilizados en un producto de software, un SBOM permite a los desarrolladores, equipos de seguridad y usuarios finales comprender los componentes funcionales de su software. Esta visibilidad ayuda a identificar posibles vulnerabilidades en componentes de terceros, lo que les permite parchearse o reemplazarse antes de que se materialicen los problemas. También ayuda a rastrear el origen de los componentes de software, lo que permite garantizar que provengan de fuentes de confianza.

Después de una divulgación de vulnerabilidad o un incidente de seguridad, un SBOM admite una respuesta y reparación rápidas. Los equipos de seguridad pueden determinar rápidamente si su software se ve afectado por una vulnerabilidad revelada en un componente en particular y tomar las medidas adecuadas.

Un SBOM es una herramienta fundamental para gestionar y garantizar la seguridad de la cadena de suministro de software, ya que contribuye a un enfoque más proactivo e informado para identificar y gestionar posibles problemas de la cadena de suministro de software.

Análisis de dependencia y herramientas SBOM
La verificación de dependencias de OWASP es una herramienta de análisis de composición de software (SCA) con la que se identifican las dependencias del proyecto y se verifica si hay alguna vulnerabilidad conocida y divulgada públicamente asociada con ellas. Esta herramienta es muy útil para crear una lista de materiales de software (SBOM), aunque no es su función principal.

Con la verificación de dependencias se puede generar un informe que detalle todas las bibliotecas y componentes utilizados en un proyecto de software y sus respectivas versiones. Este tipo de informe sirve como base para crear un SBOM, ya que enumera todos los componentes de los que depende el software. Además, el informe incluye vulnerabilidades conocidas asociadas con estos componentes, una valiosa adición a un SBOM desde una perspectiva de seguridad.

La verificación de dependencias está diseñada principalmente para ayudar a detectar componentes obsoletos o vulnerables y no proporciona toda la información que por lo general se incluye en un SBOM, como la información de licencia o una lista completa de todos los subcomponentes.

Para un SBOM más completo que incluya información adicional, otras herramientas como el rastreo de dependencias de OWASP utilizan la salida de verificación de dependencias u otras herramientas SBOM específicas que siguen los estándares SPDX u OWASP CycloneDX. El SPDX (Intercambio de datos de paquetes de software) es un estándar abierto para comunicar información de la lista de materiales de software, incluida la identificación de componentes, licencias y referencias de seguridad. CycloneDX es una especificación ligera diseñada para proporcionar una forma más ágil de compartir y analizar datos SBOM.
