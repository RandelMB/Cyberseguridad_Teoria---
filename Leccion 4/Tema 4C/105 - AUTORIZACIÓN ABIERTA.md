


Muchas nubes públicas utilizan interfaces de programación de aplicaciones (API) basadas en Representational State Transfer (REST) [transferencia de estado representacional (REST)] en lugar de SOAP. Estas se llaman API RESTful. Cuando SOAP es un protocolo estrictamente especificado, REST es un marco arquitectónico más flexible. Esto permite al proveedor de servicios contar con más opciones sobre los elementos de implementación. En comparación con SOAP y SAML, hay un mejor soporte para aplicaciones móviles. 

La autenticación y la autorización para una API RESTful suele implementarse mediante el protocolo de Open Authorization (OAuth) [autorización abierta (OAuth)]. OAuth está diseñado para facilitar el intercambio de información (recursos) dentro de un perfil de usuario entre sitios. El usuario crea una cuenta protegida con contraseña en un proveedor de identidad (IdP). El usuario puede vincular esa identidad a un sitio de consumo de OAuth sin dar la contraseña al sitio de consumo. Un usuario (propietario del recurso) puede otorgar autorización a un cliente de OAuth para acceder a alguna parte de su cuenta. Un cliente en este contexto es una aplicación o un sitio de consumo.

La cuenta de usuario está alojada en uno o varios servidores de recursos. Un servidor de recursos se denomina servidor API porque aloja las funciones que permiten a los clientes de OAuth (sitios de consumo y aplicaciones móviles) acceder a los atributos del usuario. Un servidor de autorización procesa las solicitudes de autorización. Un único servidor de autorización puede administrar múltiples servidores de recursos; de igual manera, el servidor de recursos y el de autorización podrían ser la misma instancia de servidor.

La aplicación o el servicio cliente debe registrarse en el servidor de autorización. Como parte de este proceso, el cliente registra una URL de redireccionamiento, que es el punto de conexión que procesará los tokens de autorización. El registro también proporciona al cliente un ID y un secreto. El ID se puede exponer públicamente, pero el secreto debe mantenerse confidencial entre el cliente y el servidor de autorización. Cuando la aplicación cliente solicita autorización, el usuario aprueba que el servidor de autorización otorgue la solicitud a través de un método apropiado. OAuth admite varios tipos de concesión, o flujos, para su uso en diferentes contextos, como de servidor a servidor o de aplicación móvil a servidor. Según el tipo de flujo, el cliente terminará con un token de acceso validado por el servidor de autorización. El cliente presenta el token de acceso al servidor de recursos, que luego acepta la solicitud del recurso si el token es válido.

OAuth utiliza el formato de tokens web (JWT) de JavaScript Object Notation (JSON) [notación de objetos en JavaScript (JSON)] para datos de reclamos. Los JWT se pueden pasar como cadenas codificadas en Base64 en URL y encabezados HTTP, y se pueden firmar digitalmente para su autenticación e integridad.
