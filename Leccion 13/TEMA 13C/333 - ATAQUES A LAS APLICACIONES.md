Un ataque a la aplicación tiene como objetivo una vulnerabilidad en el sistema operativo o software de la aplicación. Una vulnerabilidad de la aplicación es una falla del diseño que puede hacer que el sistema de seguridad de la aplicación sea eludido o puede ocasionar que la aplicación se bloquee. En términos generales, hay dos situaciones principales para los ataques a las aplicaciones:

Vulnerar el sistema operativo o las aplicaciones de terceros en un host de red explotando troyanos, archivos adjuntos maliciosos o vulnerabilidades del navegador. Esto permite que el actor de amenaza obtenga un punto de apoyo en una red local.
Vulnerar la seguridad de un sitio web o aplicación web. Esto permite que un actor de amenaza obtenga el control de un host web y robe datos de él o lo use para intentar penetrar más en la red.
Un mayor número de bloqueos y errores de la aplicación podría proporcionar un indicador general de que un actor de amenaza está intentando explotar una vulnerabilidad en un servicio de red, sistema operativo o aplicación de escritorio, o aplicación web. Los errores pueden registrarse en un registro del sistema o en un registro específico de la aplicación, según la naturaleza del software y el tipo de evento de falla. La utilización anómala de la CPU, la memoria, el almacenamiento o la red también puede ser un indicador de ataque a la aplicación. Sin embargo, estos indicadores también pueden tener múltiples causas no maliciosas, por lo que es importante correlacionarlos con factores que identifiquen tipos específicos de ataques a la aplicación.

Escalada de privilegios
El propósito de la mayoría de los ataques a aplicación es permitir que el actor de amenaza ejecute su propio código en el sistema. Esto se denomina ejecución de código arbitrario. Cuando el código se transmite de una máquina a otra, puede denominarse ejecución remota de código. Por lo general, el código estaría diseñado para instalar algún tipo de puerta trasera o para desactivar el sistema de alguna manera.

Una aplicación o proceso debe tener privilegios para leer y escribir datos y ejecutar funciones. Dependiendo de cómo esté escrito el software, un proceso puede ejecutarse a través de una cuenta del sistema, la cuenta del usuario conectado o una cuenta designada. Si una vulnerabilidad de software funciona, el atacante puede ejecutar código arbitrario con el mismo nivel de privilegio que el proceso explotado. Existen dos tipos principales de escalamiento de privilegios::

El escalamiento (o elevación) de privilegios vertical es cuando un usuario o aplicación puede acceder a la funcionalidad o los datos que no deberían estar disponibles para él. Por ejemplo, un proceso puede ejecutarse con privilegios de administrador local, pero una vulnerabilidad permite que el código arbitrario se ejecute con privilegios de SISTEMA más altos.
El escalamiento de privilegios horizontal es cuando un usuario accede a la funcionalidad o los datos destinados a otro usuario. Por ejemplo, a través de un proceso que se ejecuta con privilegios de administrador local en una estación de trabajo cliente, el código arbitrario puede ejecutarse como una cuenta de dominio en un servidor de aplicación.
Sin realizar un análisis detallado del código o la ejecución del proceso en tiempo real, es el escalamiento de privilegios lo que proporciona el indicador más simple de un ataque a la aplicación. Si se configuró el registro del proceso, el registro de auditoría puede proporcionar evidencia de los intentos de escalamiento de privilegios. Estos intentos también pueden ser detectados por los agentes de respuesta a incidentes y protección de punto de conexión, que mostrarán una alerta.

Desbordamiento del búfer 
Un búfer es un área de memoria que una aplicación reserva para almacenar algún valor. La aplicación esperará que los datos se ajusten a algún tamaño o formato de valor esperado. Para explotar una vulnerabilidad de desbordamiento de búfer, el atacante pasa datos que llenan deliberadamente el búfer hasta su final y luego sobrescribe los datos en su inicio. Una de las vulnerabilidades más comunes es un desbordamiento de pila. La pila es un área de memoria utilizada por una subrutina de programa. Incluye una dirección de devolución, que es la ubicación del programa que llamó a la subrutina. Un atacante podría usar un desbordamiento de búfer para cambiar la dirección de devolución, lo que permitiría al atacante ejecutar código arbitrario en el sistema. 

Los sistemas operativos utilizan mecanismos como la aleatorización del diseño del espacio de direcciones (ASLR) y la prevención de la ejecución de datos (DEP) para mitigar los riesgos del desbordamiento del búfer. Los intentos fallidos de desbordamiento del búfer pueden identificarse mediante bloqueos frecuentes del proceso y otras anomalías.
